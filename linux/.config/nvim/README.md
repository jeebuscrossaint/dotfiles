# Neovim config

A modular, beginner-friendly Neovim setup. Leader key is **Space**.
The theme is generated by [`coat`](https://github.com/jeebuscrossaint/coat) —
whenever you `coat apply` a new scheme, Neovim follows.

## First time using Neovim? Read this.

Neovim is **modal** — keys do different things depending on the mode you're in.

| Mode       | How to enter          | What it's for                    |
| ---------- | --------------------- | -------------------------------- |
| **Normal** | `<Esc>` (the default) | moving around, running commands  |
| **Insert** | `i` (or `a`, `o`)     | actually typing text             |
| **Visual** | `v` (or `V` for lines)| selecting text                   |
| **Command**| `:`                   | running commands like `:w`       |

**The golden rule:** when in doubt, press `<Esc>` to get back to Normal mode.

### The absolute minimum to survive

- `i` — start typing (insert mode). `<Esc>` — stop typing.
- Move: `h` left, `j` down, `k` up, `l` right (or just use arrow keys at first).
- `:w` + Enter — save. `:q` + Enter — quit. `:wq` — save and quit. `:q!` — quit, discard changes.
- `u` — undo. `<Ctrl>r` — redo.
- `dd` — delete a line. `yy` — copy ("yank") a line. `p` — paste.
- `/word` + Enter — search for "word". `n` — next match, `N` — previous.

### The trick that makes learning easy

Press **Space** and *wait half a second*. A menu pops up (which-key) showing
every shortcut and what it does. You never have to memorize — just explore.

Also run `:Tutor` for the official 30-minute interactive Neovim tutorial. It's
genuinely the best way to get the motions into your fingers.

## Debugging (nvim-dap)

Set breakpoints and step through your program with a graphical UI — like VS
Code's debugger. Adapters are already installed: **codelldb** (C/C++/CUDA host
code) and **debugpy** (Python).

Typical workflow:

1. **C/C++/CUDA:** compile with debug info first, e.g.
   `g++ -g -O0 main.cpp -o main`  (or `nvcc -g -G main.cu -o main`).
   **Python:** nothing to build.
2. Put your cursor on a line and press `<leader>db` to set a breakpoint (● in the gutter).
3. Press `<F5>` to start. For C/C++/CUDA it asks for the executable path
   (type the path to the binary you just compiled). The UI panels open
   automatically: variables, call stack, breakpoints, watches, and a console.
4. Step with `<F10>` (over), `<F11>` (into), `<F12>` (out). Hover a variable
   with `<leader>de` to evaluate it. `<F5>` again continues; `<S-F5>` stops.

| Key          | Action                    |
| ------------ | ------------------------- |
| `<F5>`       | start / continue          |
| `<S-F5>`     | terminate                 |
| `<F10>`      | step over                 |
| `<F11>`      | step into                 |
| `<F12>`      | step out                  |
| `<leader>db` | toggle breakpoint         |
| `<leader>dB` | conditional breakpoint    |
| `<leader>du` | toggle the debugger UI    |
| `<leader>de` | evaluate expression       |
| `<leader>dr` | open the debug REPL       |

> CUDA note: this debugs the **host (CPU)** side of a `.cu` program. Stepping
> inside GPU kernels needs `cuda-gdb` — a separate setup we can add later.

| ------------ | -------------- | -------------- | -------------------- |
| C / C++      | clangd         | clang-format   | `.c .cpp .h .hpp`    |

### C / C++ / CUDA tips (clangd)

clangd is only as smart as its knowledge of your compile flags, which it reads
from a **`compile_commands.json`**. With **CMake** this is fully automatic:

```sh
cmake -B build      # that's it
```

Your fish config sets `CMAKE_EXPORT_COMPILE_COMMANDS=ON` globally
(`~/.config/fish/conf.d/cmake.fish`), so every CMake configure writes
`build/compile_commands.json`, and clangd auto-discovers it in the `build/`
dir — no symlink, no per-project setup. (Not using CMake? `bear -- make` also
produces one.)

For **CUDA**, clangd parses `.cu` files as C++/CUDA; the same
`compile_commands.json` makes `<cuda_runtime.h>` and kernels resolve cleanly.

You'll want the toolchains installed system-wide: `sudo pacman -S gcc clang`
for C++, and the `cuda` package (+ NVIDIA driver) for CUDA.

### Python tips

- **basedpyright** does type-checking and navigation; **ruff** does linting and
  formatting (on save). Type mode is set to `"standard"` in `lua/plugins/lsp.lua`
  — bump it to `"strict"` when you want the training wheels off.
- For per-project virtualenvs, launch `nvim` from the activated venv (or point
  basedpyright at it via a `pyrightconfig.json` with `"venvPath"`/`"venv"`).

## Adding another language later

1. `:Mason`, find the server (e.g. `rust_analyzer`), press `i` to install.
2. Add it to the `servers` list in `lua/plugins/lsp.lua`.
3. (Optional) add a formatter in `lua/plugins/formatting.lua`.
4. Treesitter grabs the syntax parser automatically the first time you open a file.
